介绍一下MySQL索引原理B+树

前言

开始之前介绍一下磁盘存储原理，我们都知道磁盘是由多个盘片组成，这里我们针对于一个盘片进行分析

数据库存储的数据是如何定位的呢？

答案是通过磁道(每一个圆圈) + 扇区(圆圈上的某一部分)

磁盘中最小的存储单位扇区是512字节

重点

以下是我们将要插入数据库中的数据

假设ID占据8个字节，name占据40个字节，no占据8个字节，address占据72个字节，最终一条数据占据128个字节

因为是800条数据，\(800 * 128=102400\)，因为刚才说的最小存储单位是512，\(102400 / 512 = 200\)，如果我们想要读取其中一条数据，最坏的情况是200次

这个时候索引就出现了，索引保留的字段有数据中的ID以及新创建的地址，用来标记是哪一条数据

同理，地址和ID类型一样都是整型，也占用8个字节，即每一条索引占据16个字节，同上面一样因为是800条数据嘛，对应800个索引，计算次数\(800 * 16 / 512 =25\)，200次被压缩成25次

还有没有更快的方法？

必然是有的，刚才我们创建的是一级索引，这里接着创建二级索引，采用分组的方式，将每8个分为一组，这样我们二级索引只需要100个

次数计算等于\(25 / 8=4\)

我们能不能扩大每一组中索引的个数呢？

由于一个扇区512个字节，我们将其列为一个页，我们需要查看一个扇区可以存多少条数据，计算512 / 16(每一条索引占用字节数量) = 32，即每32个一组，二级索引的数量为\(800 / 32 =25\)，根据\(25 * 16=400\)，正常一个扇区最大字节数为512，也就是说只需要一次IO就可以读取到所有二级索引

最终成果图如下，总共包括两次IO第一次IO找到二级索引，第二次IO找到二级索引指向的一级索引，如何在一级索引里面找值，这里使用的是二分查找也可以for循环

二级索引想要完全保存需要32个记录，又因为每一个记录对应32条数据，计算\(32 * 32=1024\)，即二级索引填满的情况下最大可以查询1024个数据

将图表倒转，我们会发现和B+树形似

为了防止误解，B+树叶子节点存储的是值

针对于B树和B+树

当我们查找33的时候，B树直接获取到33下的值，不会进入子树。而B+树会进入子树、接着拿到值

B+树叶子节点通过双向链表连接，方便快速定位

如果我们的数据量从800扩充到80000，重新计算\(80000 * 16 / 512\)，需要2500次查询，为了避免IO如此之大，我们可以接着向上分(512 / 16 = 32个条目)

计算扇区的数量：\(80000 / (32 * 32)=79\)

已知一页可以存储32条数据，我们最少需要3页，完整B+树构建如下

MySQL默认从磁盘中读16K数据块(页)，由于每一个扇区是512字节，即最少读取\(16 * 1024 / 512=32\)个扇区
